# MCP 产品化与销售打法

## Summary

卖 MCP 不应从“我做了一个 Model Context Protocol 工具”开始，而应从客户已经有预算的业务问题开始。MCP 只是交付方式，真正可销售的是更快完成某个重复工作、更少人工错误、更低信息检索成本、更稳定的 Agent 工作流，或把原本分散在工具、文档、系统里的知识变成可调用能力。

## Key Points

- 不要卖协议、插件或技术名词，卖一个具体岗位或团队的结果。
- 最容易成交的 MCP 不是通用工具，而是嵌入客户现有工作流的垂直能力。
- 早期不要做 marketplace 式泛分发，先做 3-5 个高痛点场景的手工成交。
- 价格应按节省的人力、减少的错误、缩短的交付周期或新增收入定价，而不是按代码量定价。
- MCP 的信任成本高于普通模板，因为它会连接客户的数据、工具和工作流；销售页必须说明权限边界、数据流和可回滚方式。
- 最小可卖版本应包含安装、配置、示例 prompt、操作 SOP、故障处理和一次 onboarding，而不只是代码仓库。

## Facts

- MCP 的直接买家通常不是“想买 MCP 的人”，而是已经在用 AI、但卡在数据访问、工具调用、流程自动化或知识检索上的团队。
- 客户不会因为协议标准付钱，客户为业务结果、稳定性、节省时间和降低试错成本付钱。
- 一人公司早期最适合通过服务验证 MCP 需求，再把重复配置、重复培训和重复交付部分产品化。

## Interpretations

### MCP 的三种可卖形态

| 形态 | 卖点 | 适合客户 | 价格逻辑 |
| --- | --- | --- | --- |
| 垂直 MCP 工具 | 让 Agent 能稳定访问某类数据或系统 | 已有明确工具链的团队 | 按节省时间和减少错误定价 |
| MCP 集成服务 | 帮客户把内部系统接入 Agent 工作流 | 有预算但缺实现能力的团队 | 项目制或诊断 + 实施 |
| MCP 工作流包 | MCP server + prompt + SOP + eval cases | 想快速落地 AI 流程的个人或小团队 | 产品化服务或模板包 |

早期更建议卖“集成服务 + 工作流包”，因为客户真正卡住的往往不是 MCP server 代码，而是场景定义、权限配置、prompt 设计、验收标准和员工怎么用。

### 好的 MCP 选题

优先选择满足这些条件的场景：

- 客户每周重复做很多次。
- 输入数据分散在多个地方。
- 纯聊天模型无法稳定完成。
- 人工复制粘贴容易出错。
- 结果能被客户快速验证。
- 客户已经为类似问题付过钱，例如软件、外包、顾问或人工运营。

不适合早期销售的场景：

- 只是展示“能调用 API”的 demo。
- 客户没有清晰工作流，只是想尝鲜。
- 需要大量定制权限和安全审批，但客单价很低。
- 没有可量化收益，也没有高频使用场景。

## Playbook

### 1. 先把 MCP 翻译成客户语言

不要说：

> 我做了一个 MCP，可以让 Agent 调用你的系统。

应该说：

> 我帮你把客服知识库、订单数据和内部 SOP 接到 AI 助手里，让新人可以在一个对话框里查到标准答案，并且每次回答都能追溯来源。

翻译公式：

```text
帮 [具体人群] 在 [具体场景] 里，用 AI 直接调用 [关键数据/工具]，从而 [节省时间/减少错误/提升成交/降低培训成本]。
```

### 2. 做一个可收费诊断入口

早期不要先卖年费 SaaS。先卖一个低风险诊断：

- 交付：60-90 分钟工作流诊断 + MCP 可行性方案。
- 产物：系统清单、权限边界、可自动化节点、MVP 范围、报价。
- 定价：低价或中价，用来筛选认真客户。
- 目的：确认客户是否有真实痛点、数据权限、预算和上线意愿。

诊断能避免免费陪聊，也能为后续高价实施铺路。

### 3. 把报价拆成三层

| 层级 | 产品 | 典型交付 | 定价思路 |
| --- | --- | --- | --- |
| 入门 | MCP 场景诊断 | 方案、边界、MVP 清单 | 筛选客户 |
| 标准 | MCP 集成 MVP | 一个场景、一个 server、示例 prompt、SOP | 按项目收费 |
| 高阶 | Agent 工作流落地 | 多系统集成、权限、eval、培训、维护 | 高客单服务或月费 |

这样可以避免一开始就陷入“一个 MCP 插件卖多少钱”的低价值比较。

### 4. 销售页只讲一个场景

MCP 销售页应避免写成技术说明书。最小结构：

1. 目标客户：谁最适合用。
2. 痛点场景：现在怎么低效。
3. 结果承诺：装好后能完成什么。
4. 工作流示例：用户问什么，Agent 调什么，输出什么。
5. 安全边界：读哪些数据，不读哪些数据，是否写入系统。
6. 交付内容：代码、配置、prompt、SOP、培训、维护。
7. 验收标准：如何判断项目完成。
8. 下一步：预约诊断或购买模板包。

### 5. 用案例替代功能清单

MCP 最好的销售材料不是功能表，而是前后对比：

- 之前：运营要打开 4 个系统查资料，10 分钟回答一个客户问题。
- 之后：Agent 通过 MCP 查订单、政策和 SOP，30 秒生成带来源的回复草稿。
- 价值：减少等待、降低新人培训成本、提高回复一致性。

案例越具体，越能绕开“这个协议和插件有什么区别”的低价值讨论。

## Pricing Notes

MCP 不适合只按“插件数量”定价。更合理的锚点：

- 每月节省多少人工小时。
- 减少多少错误、漏单、漏回复或返工。
- 缩短多少交付周期。
- 是否替代一部分外包或内部支持成本。
- 是否支撑一个可重复销售的业务流程。

早期可用：

```text
诊断费 + MVP 实施费 + 可选维护月费
```

等出现重复客户和重复交付后，再产品化成模板、订阅或托管服务。

## Next Actions

1. 为当前 MCP 写一句客户语言定位：帮谁、在什么场景、调用什么、产生什么结果。
2. 选一个最容易展示结果的垂直场景，不要同时卖给所有行业。
3. 做一个 1 页销售页，只讲这个场景的前后对比。
4. 设计一个可收费诊断，而不是一开始免费定制。
5. 找 5 个目标客户做手工销售，记录重复问题、反对意见和愿意付钱的触发点。
