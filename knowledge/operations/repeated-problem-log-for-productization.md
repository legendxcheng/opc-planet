# 重复问题日志驱动产品化

## Summary

一人公司的产品化机会通常不是从创始人“想做什么产品”开始，而是从客户、读者、私信和服务交付中反复出现的问题开始。重复问题日志的作用，是把零散反馈变成产品判断：哪些问题高频、强痛、愿意付费、可重复解决，哪些只是好奇、闲聊或低价值咨询。

## Key Points

- 产品化前先记录重复问题，而不是直接设计课程、模板或工具。
- 私信、评论、咨询、销售访谈和服务交付都是问题来源。
- 要记录客户原话，避免把客户语言过早翻译成创始人的专业术语。
- 高频问题不一定值得产品化，还要看付费意愿、交付难度和结果可验证性。
- 日志的目标是发现可复用解决方案，而不是堆积素材。
- 每次服务后都应沉淀问题、步骤、反对意见、案例和可复用资产。

## Details

### 最小字段

| 字段 | 说明 |
| --- | --- |
| 日期 | 记录问题出现时间，方便观察频率。 |
| 来源 | 私信、评论、访谈、咨询、服务、社群等。 |
| 人群 | 提问者是谁，处在什么业务阶段。 |
| 客户原话 | 保留原始表达，不急着改写。 |
| 问题类型 | 获客、定位、内容、销售、交付、定价、工具、心态等。 |
| 付费信号 | 是否问价格、预约、购买、复购、要求方案。 |
| 当前解法 | 这次如何回答或交付。 |
| 可复用资产 | 可沉淀为模板、清单、脚本、课程、工具或案例的部分。 |
| 下一步 | 继续观察、做服务、预售、产品化或排除。 |

### 判断是否值得产品化

一个重复问题进入产品化前，至少要通过四个检查：

1. 高频：是否在不同客户、不同场景中反复出现？
2. 强痛：不解决是否会影响收入、效率、风险或重要目标？
3. 付费：是否有人愿意为诊断、辅导、模板或工具付钱？
4. 可复用：解决步骤是否能被整理成稳定流程？

如果只有高频但没有付费意愿，它更适合做免费内容；如果付费强但高度定制，它更适合继续做高价服务；只有高频、强痛、付费、可复用同时成立时，才适合产品化。

### 从日志到产品的转换

| 日志发现 | 适合产物 |
| --- | --- |
| 客户反复问同一基础概念 | 免费内容、FAQ、入门指南 |
| 客户有同一类诊断需求 | 低价诊断、检查清单、评分表 |
| 客户需要照着做的步骤 | 模板、SOP、工作簿、课程 |
| 客户需要持续反馈 | 社群、coaching、陪跑 |
| 客户需要重复执行工具 | 自动化脚本、知识库 Agent、轻量软件 |

### 运营节奏

- 每天：记录新增问题和客户原话。
- 每周：合并同类项，标注出现次数和付费信号。
- 每月：选择一个高潜力问题做服务或预售实验。
- 每次交付后：把可复用解释、截图、步骤和案例补回日志。

重复问题日志不是研究报告，而是产品化看板。它必须连接下一步动作：做内容、做访谈、报价、交付、预售或放弃。
